# 项目定位与面试讲法

## 一句话定位

这是一个面向 AI Agent 工具调用场景的轻量级 MCP Gateway 原型系统，重点展示 MCP 服务注册、工具发现、统一调用入口、API Key 鉴权、路由转发、对话测试和基础可观测能力。

## 为什么这个项目适合找实习

它覆盖了实习面试常见的几个能力点：

- 后端 API 设计：FastAPI、REST、JSON-RPC 入口。
- 协议集成：MCP SSE 服务发现与工具调用。
- AI 应用工程：Agent 根据工具 schema 调用拼团商城业务工具。
- 安全意识：API Key 哈希存储、Bearer 鉴权、移除危险的在线写代码与重启服务入口。
- 数据建模：服务、工具、路由、会话、请求统计。
- 前端工程：React 管理控制台、对话测试、工具调用卡片。
- 测试意识：核心链路有自动化测试。

## 推荐讲法

不要把它包装成“完整企业级 MCP 网关”。更合理的说法是：

> 我做了一个轻量级 MCP Gateway 原型，用来验证 AI Agent 如何安全地发现和调用企业内部工具。系统包含 MCP 服务注册、工具同步、网关路由、API Key 鉴权、对话测试和调用统计。为了让演示更贴近业务，我把示例 MCP Server 改造成拼团商城工具服务，可以查询活动、优惠、商品排行、成团进度和用户参团历史。过程中我也修复了 API Key 只做展示、健康检查不真实、在线写代码和重启服务风险过高等问题。

这比“我做了企业级系统”更可信，也更能体现工程判断。

## 当前已经完成的核心链路

```text
注册 MCP Server
  -> 同步工具 schema
  -> 创建 API Key
  -> 配置路由规则
  -> /gateway/{path} 接收 JSON-RPC 请求
  -> 鉴权
  -> 路由到目标 MCP Server
  -> 返回工具结果
  -> 记录调用统计
```

对话测试链路：

```text
用户输入
  -> 后端创建/更新聊天会话
  -> 加载历史消息
  -> LLM 选择工具
  -> MCP Client 调用工具
  -> 前端 SSE 展示工具调用过程
  -> 保存 assistant 回复
```

示例业务工具链路：

```text
real-mall-mcp
  -> 连接已有 group_buy_market MySQL
  -> 暴露拼团活动、优惠、队伍、订单、商品排行等 MCP 工具
  -> Agent 将多个工具结果汇总成运营分析回答
```

## 已主动修正的设计问题

### 1. 自动部署风险

早期版本允许后端把工具代码写入 MCP 服务源码并重启进程。这个能力虽然看起来像“低代码部署”，但安全风险太高。当前已经移除相关后端接口和前端入口，工具实现由 MCP 服务自身维护，控制台只负责同步、查看、启停和删除工具记录。

### 2. API Key 和路由规则空转

早期版本有 API Key 和路由规则的 CRUD，但没有真实参与请求链路。当前新增 `/gateway/{path}` 入口，API Key 和路由规则已经参与 `tools/list` 与 `tools/call`。

### 3. 健康检查不真实

早期版本健康检查会直接把服务标记为 online。当前健康检查会实际连接 MCP 服务，同步工具成功才标记 online，失败会标记 offline。

### 4. 对话会话未持久化

早期版本聊天表存在但不写入。当前会保存用户消息、assistant 消息、工具调用信息，并在后续请求中传递历史上下文。

### 5. 前端请求硬编码

早期版本前端直接请求 `http://localhost:8000`。当前改为请求 `/api/chat/send`，由 Vite 代理转发，便于本地和部署环境切换。

### 6. 示例工具与业务库不匹配

早期版本 `real-mall-mcp` 混合了少量真实查询和大量模拟工具，无法很好体现拼团商城业务。当前工具集已收敛为拼团场景：活动列表、活动详情、成团进度、销售分析、商品排行、优惠效果、未成团队伍和用户参团历史，并且只读取真实 `group_buy_market` 数据库。

## 仍然不是企业级的部分

这些不是缺陷，而是学生项目的合理边界：

- 没有接企业 SSO / OAuth / OIDC。
- 没有多租户隔离。
- 没有 Redis 限流和分布式锁。
- 没有完整策略引擎和 DLP。
- 没有 OpenTelemetry 全链路追踪。
- 没有 Kubernetes 部署编排。
- 前端包体积还可以继续拆分优化。

面试时可以主动说明：这个项目是“轻量级原型”，不是生产系统；重点是把关键链路跑通并体现安全与工程意识。

## 后续可扩展方向

如果继续增强，推荐按这个顺序：

1. 完善 README 和演示脚本。
2. 增加调用日志持久化表，而不是只存内存日志。
3. 为 `/gateway/{path}` 增加更完整的 JSON-RPC 错误码和 schema 校验。
4. 增加简单限流，例如按 API Key 每分钟限制请求数。
5. 增加工具调用审计详情页。
6. 对前端图表库做动态 import，减少构建包体积。
